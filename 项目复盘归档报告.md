- 正文 ≥ 3000 字

## 五、流水线设计经验沉淀

### 5.1 做得好的实践（可复用到下次项目）

1. **prompt + config 双文件机制**
 - `*_prompt.md` 给模型自然语言指令
 - `*_config.json` 给下游/审计程序结构化元数据
 - **价值**：人类与机器都可读，避免"提示词漂移"

2. **明确的上下游契约**
 - 每个 phase 的 config 中明确 `input.dependency_files` 和 `downstream.consumers`
 - **价值**：流水线可视化，断点可定位

3. **专家审核作为独立 phase**
 - 不是让撰写模型自审，而是用专门模型把关
 - **价值**：Phase 2 抓出了 Phase 1 的硬伤（美的-小天鹅分类错误），避免了下游全面返工

4. **占位符 + 编者注机制**
 - Phase 3 不硬塞数据，留 `【数据待填】`
 - **价值**：让 Phase 4 在拥有全局视野时统一处理勾稽

5. **生成式 + 脚本化**
 - 最终 docx 由 Python 脚本生成而非手工编辑
 - **价值**：可重复、可审计、可调整

### 5.2 可改进的设计点

1. **Phase 间数据传递无强校验**
 - 当前依赖下游模型主动读取上游文件
 - **改进**：引入 schema 校验，例如 Phase 3 输出必须包含特定结构标记

2. **没有显式的引用-参考文献对应表**
 - 正文中"潘红波、余明桂（2014）"与文献列表第 5 条的对应关系靠人工核对
 - **改进**：Phase 3 输出附 citation map

3. **图表占位符过于自由**
 - Phase 3 的 `【图表N：标题描述】` 描述详略不一
 - **改进**：固定 schema，要求每个图表占位符必须含字段 {type, x_axis, y_axis, data_keys}

### 5.3 方法论：多模型协同的核心要点

| 要点 | 落地方式 |
|------|----------|
| **角色互补而非冗余** | Kimi（结构化设计）+ GLM（领域知识）+ GPT（语言表达）+ Claude（综合质保）—— 每个模型用在最擅长的环节 |
| **审核早于撰写** | Phase 2 在 Phase 3 之前，发现错误的成本最低 |
| **总装最后做** | Phase 4 整合所有产物，最后做才能拥有全局视野 |
| **每阶段产出可读** | 即使流水线中断，任何 phase 的产出都是人类可读的中间产物 |

### 8.1 Phase 8 五模型模拟答辩
在"撰写→盲审→修订→设计回溯"之后补上"答辩"环节，五模型零角色重叠：8-A 主席(Kimi，最懂论文骨架)/8-B 会计专家(GLM)/8-C 方法论专家(Claude) 三层独立出题(共 21 题)→8-D 学生(GPT，撰稿人)逐题作答→8-E 评分官(Qwen，全程零卷入)双层评分(逐评审 60% + 跨维度 40%)。**开卷版终评 78/中等**。方法论维度上修最大(+20)，由此引出一个疑问：这是真理解还是抄答案？

### 8.2 开卷/盲答对照实验（本项目最具方法论价值的产出之一）
发现 8-D 学生当时被喂了含「考察意图/预期回答方向」的完整问题单——等于开卷应考。遂剥掉答案键重跑（盲答版）做对照：
- 盲答终评 **75/中等**；**开卷红利仅 −2.6 分（取整 −3）**，很小。
- **方法论四维开卷=盲答（零衰减）**：证明那 +20 上修是真实理解力，不是抄答案；红利只落在"数据底稿操作细节（CSV/字段映射）"与跨维度一致性。
- 结论：①多模型答辩产出的"深度"大体真实；②这是继 Phase 5 盲审、Phase 7 复盘之后，**第三个独立信源指向同一处老伤口 = 数据可复现性**。
- 方法副产物：把"答案键泄漏污染"从 bug 变成可量化工具（开卷 − 盲答 = 污染量）；评分官两阶段（冷评锁分 → 再对照）防锚定。
