# Phase Templates (Fill-in Ready) · Multi-Model Paper Pipeline

> Use together with `多模型论文流水线_playbook.md`. Each phase provides a prompt + config skeleton. Replace `{{占位符}}` with your own content. The skeleton is scaffolding, not the full text — keep the structure and gates, and expand the body yourself.

## Naming Convention and Directory Layout
- Two files per phase: `phase{N}_{模型}_prompt.md` + `phase{N}_{模型}_config.json`
- Phase outputs: `phase{N}_{角色}_{内容}.{md,json}` (human-readable + machine-readable dual files)
- Important deliverables (manuscript / score report / retrospective) always use `.md` + `.json` dual files

## Phase 0 ⚑ Topic Selection & Rule Lock-in + Pre-Commit Veto (Two Files)

> v1.1: Phase 0 was split into two files — **0-A**, where the designer establishes rules, and **0-V**, where an **independent** Veto Officer performs the theory kill-test. The two files belong to different models (0-V must never be the same person as 0-A).

### Phase 0-A Topic Selection & Rule Lock-in

```markdown
# Phase 0-A: Topic Selection & Rule Lock-in — {{模型}}
## Role
Research designer. Lock in the topic and all "publication gate" rules, blocking the integrity/recency root causes during the design phase.
## Tasks
1. Topic and research question: {{选题}}; core question: {{研究问题}}; paper type: {{实证/案例/综述}}
2. Operational definitions: define key concepts (such as "success," "performance improvement," and "valid signal") as adjudicable criteria — **lock them in; downstream post hoc rationalization is prohibited**
3. Data provenance rules (front-loaded): define the four tiers [R] real / [S] simulated / [E] estimated / [P] public fact; stipulate that [S]/[E] may not support empirical significance claims; abstract/body/figure-table definitions must be consistent
4. Literature recency rules: literature from the last five years ≥{{比例}}, core-journal proportion ≥{{比例}}
5. Weight-bearing gates: attach gates from playbook §5 by paper type (empirical → identification/event study/cash flow/robustness; case study → symmetry/operationalization/measurement effects; review → recency/reproducible search)
## Deliverables
- {{phase0a_design.md}} / .json: topic brief + operational definitions + four-tier provenance rules + literature gates + weight-bearing gate checklist
```
config difference fields: `"produces_rules": ["provenance_4level","literature_recency","operationalization","load_bearing_gates"]`, `"is_design_gate": true`

### Phase 0-V Pre-Commit Veto Officer (Theory Kill-Test, Independent Role)

```markdown
# Phase 0-V: Pre-Commit Veto — {{零卷入/红队模型}} (cannot be the 0-A designer)
## Role
Veto Officer. Before drafting or burning engineering time, first try to **kill** this topic/hypothesis. Your default position is "it does not work."
## Tasks
1. Write the core hypothesis as a **falsifiable proposition**: not "study {{X}}," but "{{X}} satisfies specific assertion {{Y}}, which can be overturned by data"
2. Write the **strongest counter-argument**: from the position that "this is doomed," present the sharpest theory kill (example: the ETF project's "DTW erases directionality"). A "this path will not work" argument that can be explained in 30 minutes must not be left for later
3. Design and run the **cheapest falsification experiment**: use the smallest cost (one script / one chart / one marginal correlation) to take the first shot
4. **Pre-register the death criterion**: write "what result means abandonment" as an adjudicable clause, **locked before obtaining the result**
## Gates (Non-Negotiable)
- If the cheapest falsification has **already failed → rule STOP**; do not enter Phase 1 and do not begin work
- If it passes → rule GO, but keep the death criterion visible throughout; stop when triggered, and do not move the goalposts
## Deliverables
- {{phase0v_veto.md}} / .json: falsifiable proposition + strongest counter-argument + cheapest falsification experiment and results + pre-registered death criterion + GO/STOP ruling
```
config difference fields: `"role": "premise_veto"`, `"must_be_independent_of": "phase0a_designer"`, `"verdict": ["GO","STOP"]`, `"pre_registered_kill_criteria": true`

## Phase 1 Design Architect

```markdown
# Phase 1: Design Plan — {{模型}}
## Role
Design architect. Produce an executable blueprint that downstream phases can build from.
## Tasks
1. Detailed outline (down to third-level headings)
2. Candidate {{案例/数据集/样本}} and inclusion logic — serving comparison/identification, not padding the count
3. Theoretical framework: every theory must **map to each {{案例/假设}}**, otherwise delete it
4. Initial literature list (satisfying Phase 0 recency rules)
5. Mark each chapter's data needs and expected provenance tier ([R]/[S]/[E]/[P])
## Deliverables
- {{phase1_design.md}} / .json
## Constraints
Comply with all rules locked in Phase 0.
```
config difference fields: `"upstream": ["phase0_design.md"]`, `"theory_must_map": true`

## Phase 2 Domain Expert Review (Before Drafting)

```markdown
# Phase 2: Domain Expert Review — {{模型}}
## Role
{{领域}} domain expert. Check hard flaws before drafting — this is the highest-value phase.
## What to Review
- Whether {{准则/法条/识别策略/数据口径}} is correct, complete, and defensible
- Whether the classification of candidate {{案例/样本}} is wrong (if this is wrong, everything downstream is wrong)
- Unsound hypotheses and unmapped theories in the Phase 1 blueprint
## Deliverables
- {{phase2_expert_review.md}} / .json: itemized issues + severity + recommendations
## Requirements
Review only domain hard flaws; do not rewrite prose style (that belongs to later blind review).
```
config difference fields: `"upstream": ["phase0_design.md","phase1_design.md"]`, `"review_layer": "domain_pre_write"`

## Phase 3 Drafter

```markdown
# Phase 3: Drafting — {{模型}}
## Role
Primary drafter. Write the main body, but **do not force-fit data** — leave placeholders for the assembly phase to handle consistently with full context.
## Placeholder Contract
- `【数据待填:{{字段}}】` / `【图表N:{{说明}}】` / `【编者注:{{给总装的提示}}】`
## Tasks
Draft according to the Phase 1 outline + the Phase 2 revised plan; citations follow the Phase 0 literature rules.
## Deliverables
- {{phase3_draft.md}} / .json
## Prohibited
Fabricating data into the body text; use placeholders wherever data is needed.
```
config difference fields: `"upstream": ["phase1_design.md","phase2_expert_review.md"]`, `"placeholder_contract": ["【数据待填】","【图表N】","【编者注】"]`

## Phase 4 Assembly / Delivery

```markdown
# Phase 4: Assembly and Delivery — {{模型}}
## Role
Assembler. You have the full context — complete placeholders, format, unify definitions, and perform final review.
## Tasks
1. Process each Phase 3 placeholder, and **honestly label provenance tiers** under the Phase 0 provenance rules (never label [S] as [R])
2. Unify definitions throughout the manuscript (indicator definitions, units, numerical reconciliation — abstract = body = figures/tables)
3. Standardize formatting, figures/tables, and references
4. Perform one final internal consistency review
## Deliverables
- {{总装稿.md}} / .json (including provenance-tier labels for every number)
## Red Line
Passing simulated/estimated data off as real sources = academic misconduct. Better to label it "demonstration" than to impersonate real data.
```
config difference fields: `"upstream": ["phase3_draft.md","phase0_design.md"]`, `"owns_full_context": true`, `"provenance_labeling": "mandatory"`

## Phase 5 Cross-Blind Review ×2 (A/B, Reverse, Independent, No Communication)

> Run twice: 5A and 5B use **two different models**, and each reviews **a domain it did not work on** (have the drafting model review the domain layer, and the domain-review model review the writing layer). The two outputs are independent and do not see each other.

```markdown
# Phase 5{{A/B}}: Cross-Blind Review — {{模型}} (reverse-reviewing {{对方领域}})
## Role
Independent blind reviewer, using {{研究生/本科毕业}} standards. You **did not participate in** the {{你被分到审的那块}} of this paper.
## What to Review (Choose One by Assignment)
- 5A: compliance/factual layer — whether {{法律/会计/案例/数据}} holds and is auditable
- 5B: academic standards layer — structure / argument / writing / literature / terminology / figures and tables
## Deliverables
- {{phase5x_review.md}} / .json: itemized comments + severity + score (out of 100) + overall verdict tier (pass / pass after revision / reject and rewrite)
## Requirements
Make independent judgments; do not guess how the other reviewer will review. Catching hard flaws takes priority over style.
```
config difference fields: `"upstream": ["{{总装稿.md}}"]`, `"review_layer": "5A合规事实 / 5B学术规范"`, `"reverse_domain": true`, `"blind_to_peer": true`

## Phase 6 Integration Ruling & Revision

```markdown
# Phase 6: Integration Ruling and Revision — {{模型}}
## Role
Integration officer (switch into a "stranger" stance; do not defend any old decision, even one you made earlier).
## Process
1. Receive the 5A/5B dual blind reviews → map opinions (same-direction / opposite / complementary)
2. Ruling rules: adopt in-domain single-party comments by default; downweight cross-boundary single-party comments; same-direction comments must be revised; opposite-direction comments require independent adjudication (basis: original text → Phase2 → Phase0 rules → academic common sense)
3. Weighted score: 5A×{{w}} + 5B×{{1-w}}
4. Produce unified revision list → revised manuscript → final sign-off (pass / pass after revision / reject and rewrite)
## Deliverables
- {{phase6_integration.md}} / .json (ruling record) + revised manuscript
```
config difference fields: `"upstream": ["phase5a_review.md","phase5b_review.md","{{总装稿.md}}"]`, `"weights": {"5A": "{{w}}", "5B": "{{1-w}}"}`, `"verdict_tiers": ["通过","修改后通过","退回重写"]`

## Phase 7 ⚑ Design Retrospective

```markdown
# Phase 7: Design Retrospective — {{模型}} (as the original designer)
## Role
The original design architect, now looking back to evaluate execution fidelity.
## Tasks
1. Final blueprint execution assessment (percentage + calibration): what was implemented, what was not, and what was a reasonable tradeoff vs design failure
2. **Trace the root cause of residual problems upstream to the design phase** — "what rule should have been established in Phase 0/1 to prevent this?"
3. Provide a residual must-fix list to return to the author
## Deliverables
- {{phase7_retrospect.md}} / .json
## Value
The most valuable output is not the score; it is "what gate should be added during the design phase next time."
```
config difference fields: `"upstream": ["全部前序产出"]`, `"root_cause_to_design": true`

### 8-Scoring (Zero-Involvement Model + Two-Stage Anti-Anchoring)
```markdown
# Phase 8-E: Scoring Integration Officer — {{零卷入模型}}
## Role
External scoring expert, zero-involvement throughout (wrote no questions, drafted no manuscript).
## Scoring Structure
Per-reviewer dimensions ({{60%}}) + cross-dimensional synthesis (logical consistency / learning growth / honest limitations / argument organization, {{40%}}) + self-check anchoring
## Two Stages (If Running Open-Book / Blind-Answer Control)
Stage 1: score cold using only the paper + question sheet + student record, and lock the score; Stage 2: only then review the control baseline and calculate the difference (open-book gain). Do not revise Stage 1.
## Deliverables
- {{phase8e_score_report.md}} / .json (including a directional reasonableness reference: defense score vs paper score are different measures; do not apply a ±X threshold mechanically)
```
config difference fields: `"pipeline_involvement": "zero"`, `"two_stage": true`, `"tier_weights": {"per_reviewer": "{{0.6}}", "cross_cutting": "{{0.4}}"}`

## Phase 9 ⚑ Post-Defense Finalization (Closed Loop)

```markdown
# Phase 9: Post-Defense Finalization — {{模型}}
## Role
Final manuscript owner. Take the **specific defects diagnosed in Phase 5/6/8 and actually repair them back into the paper**.
## Tasks
1. Summarize all unresolved specific defects (for example, inconsistent definitions, numbers that do not reconcile, overstrong wording, unmet gates)
2. For each item: revise / do not revise (state the reason if not revising)
3. Produce final manuscript vX; or, when archiving, produce a closure note (known defects + reasons for not fixing)
## Deliverables
- {{论文终稿_vX.md}} / .json or {{closure_note.md}} / .json
## Iron Rule
Diagnosis that does not land = wasted diagnosis. If this phase produces nothing, the project has not closed the loop.
```
config difference fields: `"upstream": ["phase5/6/8 全部诊断"]`, `"closes_loop": true`, `"output": "终稿vX 或 closure_note"`
