# 階段模板件（填空即用）· 多模型論文流水線

> 配合 `多模型论文流水线_playbook.md` 使用。每個階段給一套 prompt + config 骨架，把 `{{占位符}}` 換成你的內容即可。骨架是腳手架不是全文——保留結構與門禁，正文自己展開。

## 命名規範與目錄布局
- 每階段兩件：`phase{N}_{模型}_prompt.md` + `phase{N}_{模型}_config.json`
- 階段產出：`phase{N}_{角色}_{内容}.{md,json}`（人讀 + 機讀雙件）
- 重要交付（論文稿 / 成績單 / 復盤）一律 `.md` + `.json` 雙件

## 通用 config 欄位（各階段都有，下文只列差異欄位）
```json
{ "phase": "{{N}}", "model": "{{模型}}", "role": "{{角色}}", "task_id": "{{phaseN_xxx}}",
 "upstream": ["{{上游文件}}"], "deliverables": ["{{产出.md}}", "{{产出.json}}"],
 "paired_files": ["phase{{N}}_{{模型}}_prompt.md", "phase{{N}}_{{模型}}_config.json"],
 "academic_tier": "{{如:本科毕业论文}}", "created_date": "{{YYYY-MM-DD}}" }
```

## Phase 0 ⚑ 選題與規則鎖定 + 事前否決（兩件）

> v1.1：Phase 0 拆成兩件——**0-A** 設計者立規則，**0-V** 由**獨立**否決官做理論 kill-test。兩件分屬不同模型（0-V 絕不能是 0-A 本人）。

### Phase 0-A 選題與規則鎖定

```markdown
# Phase 0-A：選題與規則鎖定 — {{模型}}
## 角色
研究設計者。鎖定選題與所有"發布門禁"規則，從設計階段堵住誠信/時效病根。
## 任務
1. 選題與研究問題：{{选题}}；核心問題：{{研究问题}}；論文類型：{{实证/案例/综述}}
2. 操作化定義：把關鍵概念（如"成功""績效改善""有效信號"）定義成可判定標準——**寫死，禁止下游事後合理化**
3. 數據溯源規則（前置）：定義 [R]真實/[S]模擬/[E]估算/[P]公開事實 四級；規定 [S]/[E] 不得支撐實證顯著性結論；摘要/正文/圖表口徑必須一致
4. 文獻時效規則：近5年文獻≥{{比例}}、核心期刊佔比≥{{比例}}
5. 承重門禁：按論文類型從 playbook 第5節掛載（實證→識別/事件研究/現金流/穩健性；案例→對稱性/操作化/口徑效應；綜述→時效/檢索可復現）
## 產出
- {{phase0a_design.md}} / .json：選題書 + 操作化定義 + 溯源四級規則 + 文獻門禁 + 承重門禁清單
```
config 差異欄位：`"produces_rules": ["provenance_4level","literature_recency","operationalization","load_bearing_gates"]`, `"is_design_gate": true`

### Phase 0-V 事前否決官（理論 kill-test，獨立角色）

```markdown
# Phase 0-V：事前否決 — {{零卷入/红队模型}}（不能是 0-A 設計者）
## 角色
否決官。動筆/燒工程前，先試著把這個選題/假設**殺死**。你的預設立場是"它不成"。
## 任務
1. 把核心假設寫成**可證偽命題**：不是"研究 {{X}}"，而是"{{X}} 滿足可被數據推翻的具體斷言 {{Y}}"
2. 寫出**最強反方論證**：站在"註定不成"立場給最鋒利的理論 kill（例：ETF 項目的"DTW 抹掉方向性"）。能 30 分鐘講完的"此路不通"絕不留到後面
3. 設計並跑**最便宜的證偽實驗**：用最小成本（一段腳本/一張圖/一個邊際相關性）先打它一槍
4. **預登記死亡判據**：把"出現什麼結果就放棄"寫成可判定條款，**在拿到結果之前寫死**
## 門禁（不可談判）
- 最便宜證偽**已失敗 → 判 STOP**，不進入 Phase 1、不開工
- 透過 → 判 GO，但死亡判據全程掛著，觸線即停、不許移動球門
## 產出
- {{phase0v_veto.md}} / .json：可證偽命題 + 最強反方論證 + 最便宜證偽實驗及結果 + 預登記死亡判據 + GO/STOP 裁決
```
config 差異欄位：`"role": "premise_veto"`, `"must_be_independent_of": "phase0a_designer"`, `"verdict": ["GO","STOP"]`, `"pre_registered_kill_criteria": true`

```markdown
# Phase 1：方案設計 — {{模型}}
## 角色
方案設計師。出可執行藍圖，下游照此施工。
## 任務
1. 詳細大綱（到三級標題）
2. 候選{{案例/数据集/样本}}及入選邏輯——服務於比較/識別，不是湊數
3. 理論框架：每個理論必須能**逐{{案例/假设}}對位**，否則刪
4. 文獻清單初稿（滿足 Phase 0 時效規則）
5. 標註每章的數據需求與來源等級預期（[R]/[S]/[E]/[P]）
## 產出
- {{phase1_design.md}} / .json
## 約束
遵守 Phase 0 鎖定的全部規則。
```
config 差異欄位：`"upstream": ["phase0_design.md"]`, `"theory_must_map": true`

```markdown
# Phase 2：領域專家審覈 — {{模型}}

{{领域}}領域專家。在動筆前核硬傷——這是價值最高的一環。
```

## Phase 3 撰稿人

```markdown
# Phase 3：撰稿 — {{模型}}
## 角色
撰稿主力。寫正文，但**不硬塞數據**——留佔位符給總裝階段在全局視野下統一處理。
## 佔位符契約
- `【数据待填:{{字段}}】` / `【图表N:{{说明}}】` / `【编者注:{{给总装的提示}}】`
## 任務
按 Phase 1 大綱 + Phase 2 修正後的方案撰寫；引用按 Phase 0 文獻規則。
## 產出
- {{phase3_draft.md}} / .json
## 禁止
編造數據填進正文；凡需數據處用佔位符。
```
config 差異欄位：`"upstream": ["phase1_design.md","phase2_expert_review.md"]`, `"placeholder_contract": ["【数据待填】","【图表N】","【编者注】"]`

## Phase 4 總裝 / 交付

```markdown
# Phase 4：總裝與交付 — {{模型}}
## 角色
總裝官。你擁有全部上下文——補全佔位符、排版、統一口徑、終審。
## 任務
1. 逐個處理 Phase 3 的佔位符，按 Phase 0 溯源規則**如實標級**（絕不把 [S] 標成 [R]）
2. 全文口徑統一（指標定義、單位、數字勾稽——摘要=正文=圖表）
3. 排版、圖表、參考文獻規範
4. 終審一遍內部一致性
## 產出
- {{总装稿.md}} / .json（含每個數字的來源等級標識）
## 紅線
模擬/估算數據冒充真實來源 = 學術不端。寧可標"演示"也不冒充。
```
config 差異欄位：`"upstream": ["phase3_draft.md","phase0_design.md"]`, `"owns_full_context": true`, `"provenance_labeling": "mandatory"`

## Phase 5 交叉盲審 ×2（A/B，反向、獨立、互不溝通）

> 跑兩遍：5A 與 5B 用**兩個不同模型**，各審**自己沒做過的領域**（讓撰稿模型審領域、讓審領域的模型審寫作）。兩份獨立產出，不互相看。

```markdown
# Phase 5{{A/B}}：交叉盲審 — {{模型}}（反向審 {{对方领域}}）
## 角色
獨立盲審，按{{研究生/本科毕业}}標準。你**沒參與**這篇論文的{{你被分到审的那块}}。
## 審什麼（按分工二選一）
- 5A：合規事實層——{{法律/会计/案例/数据}}是否成立、可複核
- 5B：學術規範層——結構/論證/寫作/文獻/術語圖表
## 產出
- {{phase5x_review.md}} / .json：逐條意見 + 嚴重度 + 評分（滿分100）+ 總評檔（通過/修改後通過/退回重寫）
## 要求
獨立判斷，不揣測對方會怎麼審；抓硬傷優先於風格。
```
config 差異欄位：`"upstream": ["{{总装稿.md}}"]`, `"review_layer": "5A合规事实 / 5B学术规范"`, `"reverse_domain": true`, `"blind_to_peer": true`

## Phase 6 整合裁決與修訂

```markdown
# Phase 6：整合裁決與修訂 — {{模型}}
## 角色
整合官（切換"陌生人"姿態，不爲任何舊決策辯護，哪怕是你自己之前做的）。
## 流程
1. 接收 5A/5B 雙盲審 → 意見映射（同向/反向/互補）
2. 裁決規則：領域內單方預設採納；跨界單方降權；同向必改；反向獨立裁決（依據：原文→Phase2→Phase0規則→學術常識）
3. 加權評分：5A×{{w}} + 5B×{{1-w}}
4. 產出統一修改清單 → 修訂稿 → 終審簽字（通過/修改後通過/退回重寫）
## 產出
- {{phase6_integration.md}} / .json（裁決記錄）+ 修訂版論文稿
```
config 差異欄位：`"upstream": ["phase5a_review.md","phase5b_review.md","{{总装稿.md}}"]`, `"weights": {"5A": "{{w}}", "5B": "{{1-w}}"}`, `"verdict_tiers": ["通过","修改后通过","退回重写"]`

## Phase 7 ⚑ 設計回溯

```markdown
# Phase 7：設計回溯 — {{模型}}（原設計者身份）
## 角色
當初的方案設計者，回頭評執行度。
## 任務
1. 藍圖執行度終評（百分比 + 校準）：哪些實現了、哪些沒有、哪些是合理取捨 vs 設計失誤
2. **把殘餘問題的病根上移到設計階段**——"這本該在 Phase 0/1 立什麼規則就能避免？"
3. 給出交還作者的殘餘必改清單
## 產出
- {{phase7_retrospect.md}} / .json
## 價值
最有價值的產出不是打分，是"下次該在設計階段補什麼門禁"。
```
config 差異欄位：`"upstream": ["全部前序产出"]`, `"root_cause_to_design": true`

## Phase 8 答辯模擬（出題 ×3 / 學生 / 評分）

### 8-出題（按層複製三份：設計層主席 / 準則層 / 方法層）
```markdown
# Phase 8-{{A/B/C}}：答辯質詢 — {{模型}}（{{设计层主席 / 领域专家 / 方法论专家}}）
## 角色與邊界
{{主席=最懂论文骨架的模型，问"为什么这样设计、回头看对不对"；领域专家=问准则/方法适用；方法论专家=问实证/统计细节}}。**只問你這一層，不碰別人的領域。**
## 產出
- {{phase8x_questions.md}} / .json：每題含【問題】【考察意圖】【預期回答方向】【評分維度+滿分】
## 注意
出題人不評分；考察意圖/預期回答方向是給評分官的，**學生盲答時要剝掉**。
```

### 8-評分（零捲入模型 + 兩階段防錨定）
```markdown
# Phase 8-E：評分整合官 — {{零卷入模型}}
## 角色
校外評分專家，全程零捲入（沒出過題、沒寫過稿）。
## 評分結構
逐評審維度（佔{{60%}}）+ 跨維度綜合（邏輯一致性/學習成長/侷限誠實/論證組織，佔{{40%}}）+ 自檢錨定
## 兩階段（若做開卷/盲答對照）
Stage 1：只看論文+題單+學生記錄，冷評鎖分；Stage 2：才看對照基線，算分差（開卷紅利）。Stage 1 不回改。
## 產出
- {{phase8e_score_report.md}} / .json（含方向合理性參考：答辯分vs論文分是不同量綱，不套±X閾值）
```

## Phase 9 ⚑ 答辯後定稿（閉環）

```markdown
# Phase 9：答辯後定稿 — {{模型}}
## 角色
終稿負責人。把 Phase 5/6/8 診斷出的**具體缺陷真正改回論文**。
## 任務
1. 彙總所有未修復的具體缺陷（如口徑不一致、數字不勾稽、措辭過強、門禁未達標）
2. 逐條：改 / 不改（寫明不改的理由）
3. 產出終稿 vX；或封存時產出 closure note（已知缺陷 + 不修原因）
## 產出
- {{论文终稿_vX.md}} / .json 或 {{closure_note.md}} / .json
## 鐵規
診斷不落地 = 白診斷。本階段不產出 = 項目沒閉環。
```
config 差異欄位：`"upstream": ["phase5/6/8 全部诊断"]`, `"closes_loop": true`, `"output": "终稿vX 或 closure_note"`
