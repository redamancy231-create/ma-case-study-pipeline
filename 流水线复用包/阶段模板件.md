# 阶段模板件（填空即用）· 多模型论文流水线

> 配合 `多模型论文流水线_playbook.md` 使用。每个阶段给一套 prompt + config 骨架，把 `{{占位符}}` 换成你的内容即可。骨架是脚手架不是全文——保留结构与门禁，正文自己展开。

## 命名规范与目录布局
- 每阶段两件：`phase{N}_{模型}_prompt.md` + `phase{N}_{模型}_config.json`
- 阶段产出：`phase{N}_{角色}_{内容}.{md,json}`（人读 + 机读双件）
- 重要交付（论文稿 / 成绩单 / 复盘）一律 `.md` + `.json` 双件

## 通用 config 字段（各阶段都有，下文只列差异字段）
```json
{ "phase": "{{N}}", "model": "{{模型}}", "role": "{{角色}}", "task_id": "{{phaseN_xxx}}",
 "upstream": ["{{上游文件}}"], "deliverables": ["{{产出.md}}", "{{产出.json}}"],
 "paired_files": ["phase{{N}}_{{模型}}_prompt.md", "phase{{N}}_{{模型}}_config.json"],
 "academic_tier": "{{如:本科毕业论文}}", "created_date": "{{YYYY-MM-DD}}" }
```

## Phase 0 ⚑ 选题与规则锁定 + 事前否决（两件）

> v1.1：Phase 0 拆成两件——**0-A** 设计者立规则，**0-V** 由**独立**否决官做理论 kill-test。两件分属不同模型（0-V 绝不能是 0-A 本人）。

### Phase 0-A 选题与规则锁定

```markdown
# Phase 0-A：选题与规则锁定 — {{模型}}
## 角色
研究设计者。锁定选题与所有"发布门禁"规则，从设计阶段堵住诚信/时效病根。
## 任务
1. 选题与研究问题：{{选题}}；核心问题：{{研究问题}}；论文类型：{{实证/案例/综述}}
2. 操作化定义：把关键概念（如"成功""绩效改善""有效信号"）定义成可判定标准——**写死，禁止下游事后合理化**
3. 数据溯源规则（前置）：定义 [R]真实/[S]模拟/[E]估算/[P]公开事实 四级；规定 [S]/[E] 不得支撑实证显著性结论；摘要/正文/图表口径必须一致
4. 文献时效规则：近5年文献≥{{比例}}、核心期刊占比≥{{比例}}
5. 承重门禁：按论文类型从 playbook 第5节挂载（实证→识别/事件研究/现金流/稳健性；案例→对称性/操作化/口径效应；综述→时效/检索可复现）
## 产出
- {{phase0a_design.md}} / .json：选题书 + 操作化定义 + 溯源四级规则 + 文献门禁 + 承重门禁清单
```
config 差异字段：`"produces_rules": ["provenance_4level","literature_recency","operationalization","load_bearing_gates"]`, `"is_design_gate": true`

### Phase 0-V 事前否决官（理论 kill-test，独立角色）

```markdown
# Phase 0-V：事前否决 — {{零卷入/红队模型}}（不能是 0-A 设计者）
## 角色
否决官。动笔/烧工程前，先试着把这个选题/假设**杀死**。你的默认立场是"它不成"。
## 任务
1. 把核心假设写成**可证伪命题**：不是"研究 {{X}}"，而是"{{X}} 满足可被数据推翻的具体断言 {{Y}}"
2. 写出**最强反方论证**：站在"注定不成"立场给最锋利的理论 kill（例：ETF 项目的"DTW 抹掉方向性"）。能 30 分钟讲完的"此路不通"绝不留到后面
3. 设计并跑**最便宜的证伪实验**：用最小成本（一段脚本/一张图/一个边际相关性）先打它一枪
4. **预登记死亡判据**：把"出现什么结果就放弃"写成可判定条款，**在拿到结果之前写死**
## 门禁（不可谈判）
- 最便宜证伪**已失败 → 判 STOP**，不进入 Phase 1、不开工
- 通过 → 判 GO，但死亡判据全程挂着，触线即停、不许移动球门
## 产出
- {{phase0v_veto.md}} / .json：可证伪命题 + 最强反方论证 + 最便宜证伪实验及结果 + 预登记死亡判据 + GO/STOP 裁决
```
config 差异字段：`"role": "premise_veto"`, `"must_be_independent_of": "phase0a_designer"`, `"verdict": ["GO","STOP"]`, `"pre_registered_kill_criteria": true`

## Phase 1 方案设计师

```markdown
# Phase 1：方案设计 — {{模型}}
## 角色
方案设计师。出可执行蓝图，下游照此施工。
## 任务
1. 详细大纲（到三级标题）
2. 候选{{案例/数据集/样本}}及入选逻辑——服务于比较/识别，不是凑数
3. 理论框架：每个理论必须能**逐{{案例/假设}}对位**，否则删
4. 文献清单初稿（满足 Phase 0 时效规则）
5. 标注每章的数据需求与来源等级预期（[R]/[S]/[E]/[P]）
## 产出
- {{phase1_design.md}} / .json
## 约束
遵守 Phase 0 锁定的全部规则。
```
config 差异字段：`"upstream": ["phase0_design.md"]`, `"theory_must_map": true`

## Phase 2 领域专家审核（撰写前）

```markdown
# Phase 2：领域专家审核 — {{模型}}
## 角色
{{领域}}领域专家。在动笔前核硬伤——这是价值最高的一环。
## 审什么
- {{准则/法条/识别策略/数据口径}}是否正确、完整、可辩护
- 候选{{案例/样本}}的分类判定有没有错（错了下游全错）
- Phase 1 蓝图里站不住的假设、对不上的理论
## 产出
- {{phase2_expert_review.md}} / .json：逐条问题 + 严重度 + 建议
## 要求
只审领域硬伤，不改写作风格（那是后面盲审的事）。
```
config 差异字段：`"upstream": ["phase0_design.md","phase1_design.md"]`, `"review_layer": "domain_pre_write"`

## Phase 3 撰稿人

```markdown
# Phase 3：撰稿 — {{模型}}
## 角色
撰稿主力。写正文，但**不硬塞数据**——留占位符给总装阶段在全局视野下统一处理。
## 占位符契约
- `【数据待填:{{字段}}】` / `【图表N:{{说明}}】` / `【编者注:{{给总装的提示}}】`
## 任务
按 Phase 1 大纲 + Phase 2 修正后的方案撰写；引用按 Phase 0 文献规则。
## 产出
- {{phase3_draft.md}} / .json
## 禁止
编造数据填进正文；凡需数据处用占位符。
```
config 差异字段：`"upstream": ["phase1_design.md","phase2_expert_review.md"]`, `"placeholder_contract": ["【数据待填】","【图表N】","【编者注】"]`

## Phase 4 总装 / 交付

```markdown
# Phase 4：总装与交付 — {{模型}}
## 角色
总装官。你拥有全部上下文——补全占位符、排版、统一口径、终审。
## 任务
1. 逐个处理 Phase 3 的占位符，按 Phase 0 溯源规则**如实标级**（绝不把 [S] 标成 [R]）
2. 全文口径统一（指标定义、单位、数字勾稽——摘要=正文=图表）
3. 排版、图表、参考文献规范
4. 终审一遍内部一致性
## 产出
- {{总装稿.md}} / .json（含每个数字的来源等级标识）
## 红线
模拟/估算数据冒充真实来源 = 学术不端。宁可标"演示"也不冒充。
```
config 差异字段：`"upstream": ["phase3_draft.md","phase0_design.md"]`, `"owns_full_context": true`, `"provenance_labeling": "mandatory"`

## Phase 5 交叉盲审 ×2（A/B，反向、独立、互不沟通）

> 跑两遍：5A 与 5B 用**两个不同模型**，各审**自己没做过的领域**（让撰稿模型审领域、让审领域的模型审写作）。两份独立产出，不互相看。

```markdown
# Phase 5{{A/B}}：交叉盲审 — {{模型}}（反向审 {{对方领域}}）
## 角色
独立盲审，按{{研究生/本科毕业}}标准。你**没参与**这篇论文的{{你被分到审的那块}}。
## 审什么（按分工二选一）
- 5A：合规事实层——{{法律/会计/案例/数据}}是否成立、可复核
- 5B：学术规范层——结构/论证/写作/文献/术语图表
## 产出
- {{phase5x_review.md}} / .json：逐条意见 + 严重度 + 评分（满分100）+ 总评档（通过/修改后通过/退回重写）
## 要求
独立判断，不揣测对方会怎么审；抓硬伤优先于风格。
```
config 差异字段：`"upstream": ["{{总装稿.md}}"]`, `"review_layer": "5A合规事实 / 5B学术规范"`, `"reverse_domain": true`, `"blind_to_peer": true`

## Phase 6 整合裁决与修订

```markdown
# Phase 6：整合裁决与修订 — {{模型}}
## 角色
整合官（切换"陌生人"姿态，不为任何旧决策辩护，哪怕是你自己之前做的）。
## 流程
1. 接收 5A/5B 双盲审 → 意见映射（同向/反向/互补）
2. 裁决规则：领域内单方默认采纳；跨界单方降权；同向必改；反向独立裁决（依据：原文→Phase2→Phase0规则→学术常识）
3. 加权评分：5A×{{w}} + 5B×{{1-w}}
4. 产出统一修改清单 → 修订稿 → 终审签字（通过/修改后通过/退回重写）
## 产出
- {{phase6_integration.md}} / .json（裁决记录）+ 修订版论文稿
```
config 差异字段：`"upstream": ["phase5a_review.md","phase5b_review.md","{{总装稿.md}}"]`, `"weights": {"5A": "{{w}}", "5B": "{{1-w}}"}`, `"verdict_tiers": ["通过","修改后通过","退回重写"]`

## Phase 7 ⚑ 设计回溯

```markdown
# Phase 7：设计回溯 — {{模型}}（原设计者身份）
## 角色
当初的方案设计者，回头评执行度。
## 任务
1. 蓝图执行度终评（百分比 + 校准）：哪些实现了、哪些没有、哪些是合理取舍 vs 设计失误
2. **把残余问题的病根上移到设计阶段**——"这本该在 Phase 0/1 立什么规则就能避免？"
3. 给出交还作者的残余必改清单
## 产出
- {{phase7_retrospect.md}} / .json
## 价值
最有价值的产出不是打分，是"下次该在设计阶段补什么门禁"。
```

## Phase 8 答辩模拟（出题 ×3 / 学生 / 评分）

### 8-出题（按层复制三份：设计层主席 / 准则层 / 方法层）
```markdown
# Phase 8-{{A/B/C}}：答辩质询 — {{模型}}（{{设计层主席 / 领域专家 / 方法论专家}}）
## 角色与边界
{{主席=最懂论文骨架的模型，问"为什么这样设计、回头看对不对"；领域专家=问准则/方法适用；方法论专家=问实证/统计细节}}。**只问你这一层，不碰别人的领域。**
## 产出
- {{phase8x_questions.md}} / .json：每题含【问题】【考察意图】【预期回答方向】【评分维度+满分】
## 注意
出题人不评分；考察意图/预期回答方向是给评分官的，**学生盲答时要剥掉**。
```
config 差异字段：`"layer": "设计/准则/方法"`, `"domain_non_overlap": true`, `"is_questioner": true`

### 8-学生（先生成"盲答题单"：把三份问题单剥成纯题干）
```markdown
# Phase 8-D：答辩学生 — {{撰稿模型}}
## 角色
论文作者（Phase 3 撰稿人）。诚实但有准备：该自信自信，该认局限认局限并给改进方向。
## 输入（严守）
只给：论文终稿 + **纯题干** {{phase8d_blind_question_sheet.md}}。**严禁**喂含【考察意图/预期回答方向】的原始问题单。
## 产出
- {{phase8_student_record.md}} / .json：开场陈述 + 逐题作答 + 总结陈述
```
config 差异字段：`"FORBIDDEN_INPUTS": ["phase8a/b/c_questions.md（含答案键）"]`, `"input": ["论文终稿","phase8d_blind_question_sheet.md"]`

### 8-评分（零卷入模型 + 两阶段防锚定）
```markdown
# Phase 8-E：评分整合官 — {{零卷入模型}}
## 角色
校外评分专家，全程零卷入（没出过题、没写过稿）。
## 评分结构
逐评审维度（占{{60%}}）+ 跨维度综合（逻辑一致性/学习成长/局限诚实/论证组织，占{{40%}}）+ 自检锚定
## 两阶段（若做开卷/盲答对照）
Stage 1：只看论文+题单+学生记录，冷评锁分；Stage 2：才看对照基线，算分差（开卷红利）。Stage 1 不回改。
## 产出
- {{phase8e_score_report.md}} / .json（含方向合理性参考：答辩分vs论文分是不同量纲，不套±X阈值）
```
config 差异字段：`"pipeline_involvement": "zero"`, `"two_stage": true`, `"tier_weights": {"per_reviewer": "{{0.6}}", "cross_cutting": "{{0.4}}"}`

## Phase 9 ⚑ 答辩后定稿（闭环）

```markdown
# Phase 9：答辩后定稿 — {{模型}}
## 角色
终稿负责人。把 Phase 5/6/8 诊断出的**具体缺陷真正改回论文**。
## 任务
1. 汇总所有未修复的具体缺陷（如口径不一致、数字不勾稽、措辞过强、门禁未达标）
2. 逐条：改 / 不改（写明不改的理由）
3. 产出终稿 vX；或封存时产出 closure note（已知缺陷 + 不修原因）
## 产出
- {{论文终稿_vX.md}} / .json 或 {{closure_note.md}} / .json
## 铁规
诊断不落地 = 白诊断。本阶段不产出 = 项目没闭环。
```
config 差异字段：`"upstream": ["phase5/6/8 全部诊断"]`, `"closes_loop": true`, `"output": "终稿vX 或 closure_note"`
