## 一、换领域复用（保留流水线骨架）

这是 fork 最直接的价值——八阶段流水线本身是领域无关的。当前项目以 M&A 案例研究为"测试用例"，但 Phase 0-9 的结构可以原封不动移植到任何学术写作任务。

### 1.1 法学案例研究

| 需要替换 | 保留 |
|---------|------|
| 论文正文（案例研究 v2） | 流水线结构、五条铁律、双文件机制 |
| Phase 1 设计文档中的会计术语 | 复用包 playbook + 阶段模板件 |
| Phase 2 领域专家审核清单（会计准则 → 法律条文） | Phase 5 交叉盲审框架 |
| Phase 8 出题方向（会计/方法论 → 法理/程序） | Phase 8 答辩模拟（出题+作答+零卷入评分） |

**入口点**: `流水线复用包/多模型论文流水线_playbook.md` §5（按论文类型选择承重阶段）→ `阶段模板件.md` 填 `{{占位符}}`。

**关键适配**: 法学论文的"数据"是判例和法条，不是财务数据——数据溯源四级分类 [R][E][S][P] 需要重新定义（如 [R]=生效判决原文、[S]=模拟案情）。

### 1.2 金融实证论文

保留流水线结构，将 M&A 的分析框架替换为标准的实证金融框架：

- **Phase 1**: 研究设计从"制度—会计—绩效—市场"改为"假设→数据→模型→稳健性"
- **Phase 3**: 撰稿不再生成案例描述，改为生成回归结果解读
- **Phase 5A/5B**: 审查清单增加"内生性处理""样本选择偏差""异方差稳健标准误"
- **数据源**: 从 akshare 改为 CSMAR/Wind/CRSP/Compustat

**入口点**: `阶段模板件.md` → playbook §5 选择"实证研究"剖面。

**前置条件**: 需要真实金融数据库访问权限（CSMAR/RESSET/Wind 等）。

### 1.3 计算机系统论文

将流水线应用于系统设计类论文：
- **Phase 2**: 领域专家审核从会计准则改为系统架构评审
- **Phase 3**: 撰稿产物从案例分析改为系统设计文档
- **Phase 5B**: 学术规范审查改为"可复现性检查"（代码开源、实验可重复、benchmark 公平）
- **数据和图表**: benchmark 结果替代财务指标，系统架构图替代 ROE 趋势图

**关键适配**: Phase 8 答辩模拟中"会计专家"角色改为"系统架构专家"，出题方向从商誉/CAR 改为 scalable/fault-tolerant/consistent。

### 1.4 公共政策分析

- **Phase 1**: 研究设计从 M&A 框架改为政策评估框架（DID/RDD/SCM）
- **数据**: 政策文本 + 统计数据替代年报财务数据
- **Phase 2**: 审核清单从会计合规改为政策逻辑一致性
- **Phase 8**: 出题角色改为政策经济学家 + 公共管理专家

## 二、把论文"做真"

当前论文的三处已知缺陷（商誉口径矛盾、杜邦分解不自洽、CAR t 值硬编码）本质上都是**数据问题**。如果 fork 者能获取 CSMAR/Wind 的真实数据，这三处可从"方法演示"升级为可送审的学术论文。

### 2.1 修复商誉口径矛盾

- **问题**: 论文混淆了"购买日新增商誉"和"期末商誉余额"两个口径（美的-库卡案例中 259 亿 vs 289.04 亿）
- **所需数据**: CSMAR 并购重组数据库 → 购买日 PPA 明细 → 可辨认净资产公允价值 → 商誉初始确认金额
- **修复入口**: `中国上市公司并购重组成功案例研究_v2.md` §4.2.2
- **工作量**: 中（需要 PPA 数据和商誉减值测试数据）

### 2.2 修复杜邦分解不自洽

- **问题**: ROE ≠ 销售净利率 × 总资产周转率 × 权益乘数——等式不闭合
- **所需数据**: 三案例公司的真实资产负债表（总资产、净资产）× 利润表（净利润、营业收入）
- **修复入口**: `scripts/generate_docx_v2.py` 中的杜邦分析段落 + 论文 §5.3
- **工作量**: 中（数据获取 + 逐案例重算 + 文档同步更新）

### 2.3 重跑 CAR 事件研究

- **问题**: CAR t 值硬编码，非真实事件研究输出
- **所需数据**: CSMAR 日度个股收益率 + 沪深 300 指数收益率（估计窗[-120,-31] + 事件窗）
- **修复入口**: 当前仓库无事件研究代码——需新增 `scripts/event_study.py`
- **工作量**: 中-高（需实现市场模型估计 → CAR 计算 → 标准化 t 检验，参照 Brown & Warner 1985 / MacKinlay 1997）

### 2.4 补充文献综述

- **问题**: 文献综述缺少近 5 年文献（≥30%），部分英文文献缺 DOI
- **修复**: 补全 Betton/Harford/MacKinlay/Yin 的 DOI；增加 2021-2026 年 CAR 方法论和商誉减值实证研究
- **入口点**: 论文 §1.3 + `scripts/generate_docx_v2.py` `build_chapter_1()`
- **工作量**: 低-中

**综合评估**: 四项修复全部完成后，论文具备送审本科/硕士论文的基本条件。但需要投入真实数据获取成本（CSMAR 数据库订阅）和事件研究实现工作。

## 四、扩展流水线本身

当前流水线是线性的 Phase 0→9。fork 可以改造流水线拓扑和机制。

### 4.1 并行多路盲审

- **当前**: Phase 5A（合规事实层）+ Phase 5B（学术规范层）两路串行
- **扩展**: 增加 Phase 5C（方法论层）/ 5D（写作质量层），多路并行盲审
- **Phase 6 升级**: 从"双审→整合裁决"改为"N 路盲审→投票/多数裁决"
- **入口点**: `phases/phase5a_gpt5.5/` + `phases/phase5b_glm5.1/` 的 prompt+config 模式
- **关键原则（不可违反）**: 铁律 2——没有任何模型审查自己做过的环节

### 4.2 迭代回路

- **当前**: Phase 7（设计回溯）只记录教训，不触发重做
- **扩展**: Phase 7 检测到设计缺陷 → 回到 Phase 1 重新设计方案 → 重新执行后续阶段
- **挑战**: 迭代终止条件（最多 N 轮？评分收敛？人工终止？）
- **复用包影响**: playbook §3 铁律需要增加"迭代终止规则"

### 4.3 人类在环（Human-in-the-Loop）

在每个 Phase 的模型产出后插入人工审核节点：

```
Phase N: AI 产出 → _human_review.md（人工读取→批准/退回/修改）→ Phase N+1
```

- **审核节点**: README 中标注哪些 Phase 有人工闸门
- **审核记录**: 人工决策记录在 `_human_review.md` 中（批准理由/退回原因/修改要求）
- **与五条铁律的关系**: 人类可以打破铁律（如让同一人审核和评分），但必须在审核记录中标注

### 4.4 多语言并行撰稿

- **当前**: Phase 3 生成中文论文 → 事后翻译
- **扩展**: Phase 3 同时生成中英双语版本 → Phase 5 盲审同时检查双语一致性
- **入口点**: `phases/phase3_gpt5.5/phase3_gpt5.5_prompt.md` 改为双语 prompt
- **挑战**: 双语版本间的术语一致性、段落对齐、参考文献格式差异

## 五、教学/培训用途

这是最被低估的方向。这套流水线的真正价值不限于产出论文——它展示了**多个 AI 模型如何以结构化、可审计的方式协作**。

### 5.1 研究生方法课

- **用法**: 让学生扮演不同 Phase 的"模型角色"（人模拟 AI），体验交叉盲审和答辩
- **操作**: 每人领取一个 Phase 的 prompt → 用自身知识（而非 AI）完成 → 交接给下一 Phase 的同学
- **教学目标**: 理解学术生产的结构化流程、peer review 的逻辑、方法论在写作前的重要性
- **入口点**: `流水线复用包/` + phases/ 中的 prompt 文件作为"角色说明书"

### 5.2 AI 协作工作坊

- **用法**: 用本仓库作为教学案例，讲解：
  - prompt+config 双文件机制的设计原理
  - 角色槽位 vs 固定模型分配
  - 零卷入评分的独立验证价值
  - 开卷/盲答对照实验的设计逻辑
- **配套材料需求**: 工作坊 slides、hands-on 练习、讨论问题（fork 可新增 `workshop/` 目录）

### 5.3 企业内训

- **场景**: 把 M&A 案例替换为公司内部的报告/提案
- **流程**: 市场分析报告 → Phase 2 行业专家审核 → Phase 3 撰稿 → Phase 5 合规+规范双审
- **价值**: 展示如何用多模型协作提升内部文档质量和一致性
- **关键适配**: 将 Phase 的领域术语替换为企业内部术语

### 5.4 提示词工程教学

Phase 1-8 的 prompt 文件是真实的生产级提示词范例：
- Phase 1 prompt: 研究设计方案生成（结构化输出、约束条件）
- Phase 5A prompt: 合规审查（检查清单、引用验证）
- Phase 8 prompt: 答辩出题（角色扮演、评分标准）
- 比教科书中的简单示例更完整、更真实

## 六、方法论改进方向

### 6.1 跨项目复用验证

- **问题**: 当前流水线只在 M&A 案例研究上验证过一次——样本量 n=1
- **方向**: 将流水线应用到 3-5 个不同领域的学术写作任务，收集以下数据：
  - 各领域的 Phase 5 盲审通过率
  - Phase 8 开卷 vs 盲答的分数差异模式
  - 不同领域对"领域专家审核"（Phase 2）的依赖程度
- **产出**: 一篇关于"多模型学术流水线跨领域泛化能力"的方法论论文
- **入口点**: `流水线复用包/` 直接移植到新项目

### 6.2 模型组合 A/B 测试

- **问题**: 当前使用 Kimi/GLM/GPT/Claude/Qwen 五个模型，但未系统测试不同组合
- **方向**: 固定论文题目 → 变化模型组合（如 GPT+Claude vs Kimi+Qwen vs 全 GPT） → 比较 Phase 5/8 评分
- **控制变量**: 论文题目、Phase 配置、评分标准不变
- **产出**: 量化"模型多样性"对学术产出质量的实际效应量
- **前置条件**: 需要 API 访问多个模型

### 6.3 流水线脆弱性测试

- **方向**: 对流水线做系统性压力测试：
  - 在 Phase 1 注入设计缺陷 → 观察哪个 Phase 最先检测到
  - Phase 3 故意生成低质量稿件 → 观察 Phase 5 盲审能否拦截
  - Phase 8 出题人故意问无关问题 → 观察评分官能否识别
- **价值**: 量化流水线各阶段的"防御深度"
- **产出**: 流水线安全边界分析报告

### 6.4 审查方法论泛化

本项目的核心方法论价值在于其**审查流程**（多轮跨后端审查 + 0 回归 + 交叉双盲）。Fork 可将这套 SOP 应用于：
- 其他 AI 辅助学术写作项目的质量门禁
- CI/CD 风格的"学术写作 pipeline"（GitHub Actions 自动运行盲审）
- 建立"AI 辅助学术生产 checklist"通用框架

## 七、语言与翻译方向

### 7.1 扩展为更多语言

当前三语言（zh-CN + en + zh-Hant）。可扩展：
- **日语**: 学术写作在日本有独特的"論文作法"传统，适配流水线
- **法语/德语**: 欧洲学术写作规范
- **关键**: 不只是翻译——各语言版本的 Phase 5B（学术规范层）的检查清单需要适配当地学术规范

### 7.2 翻译流程工具化

- **当前**: OpenCC 转换 + GPT 校对的人工流程
- **方向**: 将三语言翻译流程封装为 CI/CD（类似 translation-drift.yml 但自动化翻译更新）
- **入口点**: `.github/workflows/translation-drift.yml`
- **注意**: 当前 workflow 只检测漂移，不自动翻译——自动翻译的质量需要人工审核闸门

## 九、AI 协作元研究

这个方向不是修改仓库本身，而是**以本仓库为研究对象**做元层次的分析。

### 9.1 多模型协作中的"对齐税"

- **问题**: 多个模型协作时，prompt 交接的损耗有多大？
- **分析对象**: 本仓库 phases/ 中的 prompt 和 output——Phase 1 的意图有多少被 Phase 3 准确执行？
- **方法**: 对每个 Phase 的 prompt（输入）和 output（输出）做意图对齐分析
- **产出**: "对齐税"量化——多模型协作中信息损失的模式和量级

### 9.2 开卷/盲答对照的方法论意义

- **当前发现**: 开卷 78 / 盲答 75，方法论维度零衰减
- **可扩展**: 在不同领域、不同模型组合下重复对照实验 → 检验"方法论维度零衰减"的泛化性
- **关联**: Phase 8 的完整答辩记录（`phases/phase8/`）是宝贵的原始数据

### 9.3 Claude/GPT/DeepSeek 学术写作风格比较

- **对象**: Phase 3（撰稿）、Phase 4（总装）、Phase 8（学生作答）由不同模型执行
- **分析维度**: 论证深度、引用密度、术语一致性、对模糊指令的容忍度
- **数据**: phases/ 中的 prompt+config+output 三元组

## 关键约束与注意点

### 许可证边界

1. **分层许可**: 文档/数据/图表 → CC BY 4.0；代码（`scripts/*.py`、`.githooks/pre-push`）→ MIT。Fork 新增内容可选择自己的许可，但须保留原始文件的许可声明。

2. **数据许可**: 当前论文中的财务数据来自 akshare（东方财富）——akshare 的数据再分发权利需单独核查。fork 若新增数据源（CSMAR/Wind/Bloomberg），须遵守该数据源的许可条款。

### CLOSED-FINAL 状态

3. **当前仓库不再更新**: 本项目已封存。Fork 可以自由修改 fork 版本，但本仓库的 phase 产物不会被 upstream 修改。

4. **阶段产物不可修改原则**: 在本仓库中，phases/ 是历史快照——fork 可以修改自己的副本，但如果 fork 目的是"修正论文缺陷"，需在 README 和 CLAUDE.md 中明确标注与原版的差异。

### 五条铁律不可妥协

5. **铁律 2（零自审）**: 没有任何模型审查/评分自己做过的环节——这是流水线最核心的约束，fork 的任何扩展都必须遵守。

6. **铁律 5（诚信红线）**: 模拟数据绝不标成真实来源——fork 若补数据，须将当前 [S] 标识的数据替换为真实数据后移除 [S] 标记，而非简单删除标记。

### 版本轴说明

7. **四版本轴**: 本仓库存在 GitHub Release v1.0 / 论文修订 v2 / 归档编号 v3.0 CLOSED-FINAL / 复用包 v1.2 四个独立版本号。Fork 应建立自己的版本体系，不继承这些编号。
