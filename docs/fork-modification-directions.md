## 一、换领域复用（保留流水线内核）

流水线的**方法内核**（prompt+config 双文件、角色槽位、零卷入评分、交叉盲审）是领域无关的。但"原封不动移植"不现实——每个领域需要裁剪阶段数量、角色、门禁和输出格式。

### 1.1 法学案例研究

| 需要替换 | 保留 |
|---------|------|
| 论文正文（案例研究 v2） | 流水线内核（五条铁律、双文件机制、角色槽位） |
| Phase 1 设计文档中的会计术语 | 复用包 playbook + `流水线复用包/阶段模板件.md` |
| Phase 2 审核清单（会计准则 → 法律条文、法域、效力层级） | Phase 5 交叉盲审框架 |
| Phase 8 出题方向（会计/方法论 → 法理/程序/证据规则） | Phase 8 答辩模拟结构 |

**入口点**: `流水线复用包/多模型论文流水线_playbook.md` §5 → `流水线复用包/阶段模板件.md` 填 `{{占位符}}`。

**工作量: 中-高**（非"低"）。法学需处理法域、裁判层级、效力、法条版本、判例引用规则和冲突法——不只是术语替换。数据溯源四级分类 [R][E][S][P] 的核心语义应保持稳定，通过 task profile 扩展法源类型、效力、法域和 as-of-date 字段，而非随意重定义。

**前置条件**: 法学领域专家审核。

### 1.2 金融实证论文

- **Phase 1**: 研究设计从"制度—会计—绩效—市场"改为"假设 → 数据 → 模型 → 稳健性"
- **Phase 3**: 撰稿不再生成案例描述，改为生成回归结果解读
- **Phase 5A/5B**: 审查清单增加"内生性处理""样本选择偏差""异方差稳健标准误"
- **数据源**: 从 akshare 改为 CSMAR/Wind/CRSP/Compustat

**前置条件**: 真实金融数据库访问权限（CSMAR/RESSET/Wind 等）。

### 1.3 计算机系统论文

- **Phase 2**: 领域专家审核从会计准则改为系统架构评审
- **Phase 5B**: 学术规范审查改为"可复现性检查"（代码开源、实验可重复、benchmark 公平）
- **Phase 8**: 出题角色从会计专家改为系统架构专家

### 1.4 公共政策分析

- **Phase 1**: 研究设计改为政策评估框架（DID/RDD/SCM）
- **数据**: 政策文本 + 统计数据替代年报财务数据
- **Phase 8**: 出题角色改为政策经济学家 + 公共管理专家

## 二、把论文"做真"

当前论文有三处已知缺陷。它们的本质**不全是数据问题**——修复需要区分数据获取、计算实现和口径定义三类工作。

### 2.1 修复商誉数字不一致

- **实际情况**（v1.0 文档描述有误，v2.0 修正）: 论文正文**已**区分购买日新增商誉（约 259 亿元）和期末商誉余额（289.34 亿元）。真正的问题是**同一口径内部数值矛盾与勾稽失败**：(a) 正文/摘要写 289.34 亿，表 4 写 289.04 亿——两个数字不一致；(b) 259 + 30.21 = 289.21，不等于 289.34 或 289.04——无法勾稽；(c) 需确定 authoritative source 并统一 Markdown、JSON、生成脚本、表格和结论中的全部引用
- **修复入口**: `中国上市公司并购重组成功案例研究_v2.md` §4.2.2 + `scripts/generate_docx_v2.py` 美的数据字典
- **修复面**: Markdown → JSON → 生成脚本 → 表格 → DOCX → 摘要/结论（所有引用处同步更新）
- **工作量**: 中（需 PPA 数据或年报附注 + 全链路同步）

### 2.2 修复杜邦分解不自洽

- **实际情况**: ROE ≠ 销售净利率 × 总资产周转率 × 权益乘数。根因包含加权 ROE 与期末因子混用：4.27% × 0.78 × 2.59 = 8.63%，不等于 10.42%。需要用年报原始数据确定正确口径和计算公式，而非简单替换数字
- **修复入口**: 论文 §5.3 + `scripts/generate_docx_v2.py` 杜邦分析段落
- **修复面**: 同 §2.1——所有引用处同步
- **工作量**: 中（三公司完整财务报表 + 口径决策 + 全链路同步）

### 2.3 重跑 CAR 事件研究

- **实际情况**: CAR 数据在 `scripts/generate_docx_v2.py:145-157,1269-1273` 硬编码，**不存在事件研究实现代码**。t 值同样是硬编码数值，非统计检验输出
- **所需数据**（两条路线）: 路线 A（公开）——akshare/tushare 日度收益率 + 沪深 300；路线 B（付费）——CSMAR 日度个股收益率
- **修复入口**: 需新增 `scripts/event_study.py`（proposed——当前不存在），实现市场模型估计 → CAR 计算 → 标准化 t 检验（参照 Brown & Warner 1985 / MacKinlay 1997）
- **工作量**: 中-高（实现 + 验证 + 与硬编码值对比分析）

### 2.4 补充文献综述

- **问题**: 文献综述缺少近 5 年文献（≥30%），部分英文文献缺 DOI
- **修复**: 补全 Betton/Harford/MacKinlay/Yin 的 DOI；增加 2021-2026 年 CAR 方法论和商誉减值实证研究
- **工作量**: 低-中

**综合评估**: 四项修复**消除当前已知阻塞项，但不构成送审保证**。仓库仍存在幸存者偏差、因果识别弱和底稿缺失等深层问题；学校格式、伦理、导师审核不由修复自动满足。README 明确声明本项目为方法演示而非可送审论文——fork 若要送审需额外经领域专家和导师独立复核。

## 四、扩展流水线本身

### 4.1 并行多路盲审

- **当前**: Phase 5A（合规事实层）和 Phase 5B（学术规范层）从 Phase 4 **分叉为两条独立并行盲审**（prompt 和评分官互不沟通），在 Phase 6 汇合
- **扩展**: 增加 Phase 5C（方法论层）/ 5D（写作质量层），多路并行 → Phase 6 整合
- **裁决机制**: 多数票仅适用于可聚合偏好（风格、可读性）；**事实错误、诚信、安全或法律红线不能由多数票覆盖**——必须设置 veto/升级规则，一票触发人工裁决
- **关键原则**: 铁律 2——没有任何模型审查/评分自己做过的环节

### 4.2 迭代回路

- **当前**: Phase 7（设计回溯）只记录教训，不触发重做
- **扩展**: Phase 7 检测到设计缺陷 → 回到 Phase 1 重新设计方案
- **挑战**: 迭代终止条件（最多 N 轮？评分收敛？人工终止？）

### 4.3 人类在环（Human-in-the-Loop）

在每个 Phase 的模型产出后插入人工审核节点。**删除 v1.0 中"人类可以打破铁律"的笼统例外**——独立性风险不会因审核者是人而消失。如需例外：(a) 定义 waiver 条件；(b) 利益冲突披露；(c) 替代独立复核；(d) 审计留痕。

### 4.4 多语言并行撰稿

- **当前**: Phase 3 生成中文论文 → 事后翻译
- **扩展**: Phase 3 同时生成中英双语版本 → Phase 5 审查同时检查双语一致性

## 六、方法论改进方向

### 6.1 跨项目复用验证
将流水线应用到 3-5 个不同领域的学术写作任务（当前 n=1），收集盲审通过率和开卷/盲答差异数据。

### 6.2 模型组合 A/B 测试

- **v1.0 设计缺陷**: 单题单次实验无法区分模型多样性增益、能力差异和品牌多样性
- **修正设计**: 多任务 × 多次重复 × 随机化/配对 × 盲化 × 人类 gold set × 成本/时延记录 × 预注册 × 同模型独立实例对照

### 6.3 流水线脆弱性测试
在 Phase 1 注入设计缺陷、Phase 3 故意生成低质量稿件——量化各阶段"防御深度"。

### 6.4 审查方法论泛化
将多轮跨后端审查 + 交叉双盲 SOP 泛化为通用框架。

## 十、安全、隐私与治理（新增）

> v2.0 新增。v1.0 缺失此维度——对学术场景影响有限，但对 §4.3、§5.3、§7.2、§8 中的企业/自动翻译/CI 盲审方向至关重要。

### 10.1 数据分级与处理边界

| 数据类型 | 风险 | 处理规则 |
|---------|------|---------|
| 公开学术数据 | 低 | 标注来源、许可和获取日期 |
| 内部报告/提案（含 PII） | 高 | 发送给模型前脱敏；不在公开仓库中保留 |
| 商业秘密/受监管信息 | 极高 | 不发送给外部模型供应商；使用本地模型或人工处理 |
| 财务/法律建议 | 高 | 必须经专业人士审核并签署；标注"非正式建议" |

### 10.2 模型供应链安全

- **prompt injection**: 交叉盲审中被审稿件可能含对抗性指令 → 审查 prompt 使用独立上下文窗口
- **供应商数据保留**: 检查使用的模型 API 是否将数据用于训练
- **跨境传输**: 受监管数据须确认模型供应商的服务器地理位置

### 10.3 自动发布闸门

- 自动翻译/自动盲审 → 输出**不直接发布** → 人工审核闸门 → 签署确认 → 发布
- 法律/财务/安全结论**不通过自动流程生成和发布**

## 十一、工程化方向（新增）

> v2.0 新增。当前复用依赖手工复制 Markdown/config——多项目复用中最昂贵的配置漂移、运行恢复和证据链问题未被解决。

### 11.1 里程碑路线

| 里程碑 | 产出 | 工作量 |
|--------|------|--------|
| **M1: Validator** | JSON Schema 校验 phase config + prompt 模板 | 中 |
| **M2: Scaffolder** | CLI 一键生成新项目骨架（填 `{{占位符}}`） | 中 |
| **M3: Runner** | DAG 执行引擎（resume/retry/cache + 模型适配器） | 高 |
| **M4: Provenance** | 审计日志 + 成本/时延追踪 + 产出 manifest | 中-高 |

### 11.2 模型适配器

不绑定固定模型——通过 adapter 接口支持多供应商（OpenAI/Anthropic/DeepSeek/Kimi/GLM/Qwen），记录模型版本、参数和上下文窗口。

## 十二、反模式（不要做的事）

> v2.0 新增。基于本仓库实际踩过的坑。

1. **不要把当前固定模型分配当成跨项目最佳配置**——角色是槽位，每个项目重配
2. **不要仅删除 [S] 标记**——必须先替换为已验证真实数据，更新来源、口径和所有派生物
3. **不要用多数票覆盖事实、诚信、安全或法律红线**——设置 veto 规则
4. **不要在历史快照目录内直接重写过去产物**——模板、实验和历史证据分层存放
5. **不要在单题单次实验上宣称模型多样性的因果效应**——需要多任务、多次重复、随机化、预注册
6. **不要删除复盘与失败证据后仍声称模板经过验证**
7. **不要把敏感数据、PII 或受许可限制数据发送给未经批准的模型供应商**
8. **不要自动翻译后无人工审核直接发布**
9. **不要直接比较不同评分模型、量表或配置的分数**
10. **不要在没有 JSON Schema、版本字段和迁移策略时批量改 config**

## 十三、与 Academic Research Skills (ARS) 的互补关系

> v2.1 新增。本仓库是方法论设计手册 + 实证证据——不包含可安装的执行引擎。

[Academic Research Skills](https://github.com/Imbad0202/academic-research-skills)（ARS, v3.19.0, CC BY-NC 4.0, 作者 Cheng-I Wu）是一套 **Claude Code 学术研究技能套件**，提供可安装的 10 阶段学术流水线执行引擎（自动调度、状态追踪、诚信验证、断点续传）。两个项目在**不同抽象层**上互补：

| 维度 | ma-case-study-pipeline | ARS |
|------|----------------------|-----|
| **定位** | 方法论设计手册 + 一次性执行的实证记录 | 可安装复用的 Claude Code 技能套件 |
| **运行方式** | 阅读 playbook → 手动复制模板 → 逐阶段执行 | `claude` 对话中输入需求 → 自动调度 |
| **核心贡献** | 五条铁律、开卷/盲答对照实验、答辩模拟、[R][E][S][P]四级分类、fork 导航 | 10 阶段编排器、5 人审稿团、7 类 AI 失败模式检测、Material Passport 断点续传 |
| **状态** | CLOSED-FINAL（归档方法论） | 活跃维护（v3.19.0, 2026-07-22） |

### 互补使用建议

- **如果你想复用本仓库的方法论但不想手工执行每个阶段** → 安装 ARS 作为执行引擎，用本仓库的五条铁律和对照实验设计作为方法论参考
- **如果你用 ARS 跑出了论文但想知道方法论是否经得起实证检验** → 读本仓库的开卷/盲答对照实验（开卷 78 vs 盲答 75，方法论维度零衰减）
- **如果你在使用 ARS 但缺少答辩模拟环节** → 参考本仓库 Phase 8 的三权分离设计（出题/作答/评分由不同模型执行）
- **如果你在 ARS 中需要更细粒度的数据溯源** → 参考本仓库的 [R][E][S][P] 四级分类规范

### 两项目共同的方法论 DNA

尽管定位不同，两个项目共享多条设计原则：
- 人类在环（human-in-the-loop），不可跳过的强制检查点
- 零卷入评分/审查（不对自己做过的工作评分）
- 诚信红线不可谈判
- 失败模式可见化（ARS 的 7 类检查清单 ↔ 本仓库的 3 处已知缺陷标注 + 反模式清单）

### 许可注意

ARS 使用 CC BY-NC 4.0（限制商用），本仓库使用 CC BY 4.0 + MIT（更宽松，允许商用）。Fork 者如果将两个项目的方法论合并到自己的项目中，需注意许可兼容性。

## 关键约束与注意点

### 许可证边界

1. **分层许可**: 文档/数据/图表 → CC BY 4.0；代码（`scripts/*.py`、`.githooks/pre-push`）→ MIT。Fork 新增内容可选择自己的许可，但须保留原始文件的许可声明。
2. **数据许可**: akshare（东方财富）数据的再分发权利需单独核查。fork 新增数据源须遵守其许可条款。

### CLOSED-FINAL 状态

3. **当前仓库不再更新**。Fork 可自由修改 fork 版本。
4. **阶段产物不可修改原则**: phases/ 是历史快照——fork 若修改论文缺陷，须在 README 和 CLAUDE.md 中标注与原版差异。

### 五条铁律

5. **铁律 2（零自审）**: 没有任何模型审查/评分自己做过的环节。此条无例外——人类也不能打破独立性约束。
6. **铁律 5（诚信红线）**: 模拟数据绝不标成真实来源——fork 补数据时须替换数据 + 移除 [S] 标记，而非简单删除标记。
7. **铁律例外管理**: 若确实需要例外（如紧急修正），须：(a) 定义 waiver 条件；(b) 利益冲突披露；(c) 替代独立复核；(d) 审计留痕。

### 版本轴说明

8. **四版本轴**: GitHub Release v1.0 / 论文修订 v2 / 归档编号 v3.0 CLOSED-FINAL / 复用包 v1.2。Fork 应建立自己的版本体系，不继承这些编号。
